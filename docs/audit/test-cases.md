# Casos de Teste - Trilha de Auditoria (Audit)

Este documento define os cenários de teste E2E para validação das regras de negócio associadas ao visualizador de trilha de auditoria do sistema.

---

## 🔴 Casos de Teste Críticos

| ID / Cenário                                                        | Objetivo                                                                                                                                   | Pré-condições                                                                               | Passos                                                                                       | Resultado Esperado                                                                                                                                                                                                                                                                      |
| :------------------------------------------------------------------ | :----------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------ | :------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **CT-AUD-01**<br>Registro de auditoria na criação de pedido         | Validar que ao criar uma nova ordem de venda, um log de auditoria do tipo `CREATE_ORDER` é gerado com os dados corretos.                   | - Sistema com estado inicial.                                                               | 1. Criar um novo pedido via `/orders`.<br>2. Navegar para `/audit`.                          | - Um novo registro aparece no topo da tabela de auditoria.<br>- `actionType` = "CREATE_ORDER".<br>- `entityAffected` = "SALES_ORDER".<br>- `entityId` = ID do pedido criado.<br>- `previousState` = NULL (ou ausente).<br>- `nextState` = JSON com os dados completos do pedido criado. |
| **CT-AUD-02**<br>Registro de auditoria na transição de status       | Validar que ao transicionar o status de um pedido, um log `UPDATE_STATUS` é gerado com `previousState` e `nextState` contendo a diferença. | - Pedido `so-1` com status "Criada".                                                        | 1. Transicionar `so-1` para "Planejada" via painel de detalhes.<br>2. Navegar para `/audit`. | - Registro `UPDATE_STATUS` aparece no topo.<br>- `previousState`: JSON do pedido com status "CRIADA".<br>- `nextState`: JSON do pedido com status "PLANEJADA".<br>- `entityId` = "so-1".                                                                                                |
| **CT-AUD-03**<br>Registro de auditoria na atualização de transporte | Validar que ao alterar o tipo de transporte de um pedido, um log `UPDATE_TRANSPORT` é gerado.                                              | - Pedido com transporte "Caminhão".<br>- Cliente tem "Carreta" como alternativa autorizada. | 1. No painel de detalhes, trocar transporte para "Carreta".<br>2. Navegar para `/audit`.     | - Registro `UPDATE_TRANSPORT` aparece.<br>- `previousState`: JSON com `transportTypeId` = "trans-1".<br>- `nextState`: JSON com `transportTypeId` do novo transporte.                                                                                                                   |
| **CT-AUD-04**<br>Registro de auditoria no agendamento de entrega    | Validar que ao agendar uma entrega, um log `UPDATE_DELIVERY` é gerado.                                                                     | - Pedido com status "Planejada".                                                            | 1. Agendar entrega (data + janela).<br>2. Navegar para `/audit`.                             | - Registro `UPDATE_DELIVERY` aparece.<br>- `previousState`: JSON sem `deliveryDate`/`deliveryWindow` (ou com valores anteriores).<br>- `nextState`: JSON com `deliveryDate`, `deliveryWindow` e status "AGENDADA".                                                                      |

## 🟡 Casos de Teste Importantes

| ID / Cenário                                                               | Objetivo                                                                               | Pré-condições                                                                  | Passos                                                                                         | Resultado Esperado                                                                                                                                                                                  |
| :------------------------------------------------------------------------- | :------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **CT-AUD-05**<br>Inspeção de detalhes - Expandir linha                     | Validar que ao clicar em "Inspecionar", a linha expande exibindo os JSONs lado a lado. | - Existe pelo menos 1 registro de auditoria com `previousState` e `nextState`. | 1. Clicar em "Inspecionar" em um registro de `UPDATE_STATUS`.                                  | - Linha expande abaixo do registro.<br>- Lado esquerdo: "Estado Anterior" com JSON formatado.<br>- Lado direito: "Próximo Estado" com JSON formatado.<br>- Botão "Inspecionar" muda para "Ocultar". |
| **CT-AUD-06**<br>Ocultar detalhes - Recolher linha                         | Validar que ao clicar em "Ocultar", a linha expandida é recolhida.                     | - Uma linha de auditoria está expandida (inspecionada).                        | 1. Clicar em "Ocultar".                                                                        | - Linha expandida recolhe.<br>- Tabela volta ao estado normal.<br>- Botão volta a exibir "Inspecionar".                                                                                             |
| **CT-AUD-07**<br>Cópia de payload JSON                                     | Validar o botão "Copiar" nos painéis de Estado Anterior e Próximo Estado.              | - Linha de auditoria expandida com dados JSON.                                 | 1. Clicar em "Copiar" no painel "Estado Anterior".<br>2. Colar (Ctrl+V) em um editor de texto. | - Texto "Copiado!" aparece brevemente (1.5s).<br>- Clipboard contém o JSON formatado exatamente como exibido.                                                                                       |
| **CT-AUD-08**<br>Visualização de registro sem previousState (CREATE_ORDER) | Validar que registros de criação exibem "NULL" no lugar de `previousState`.            | - Registro de `CREATE_ORDER` existe (não tem previousState).                   | 1. Clicar em "Inspecionar" no registro de criação.                                             | - Painel "Estado Anterior" exibe: `NULL`.<br>- Painel "Próximo Estado" exibe o JSON do pedido criado.                                                                                               |

## 🔵 Casos de Teste Complementares

| ID / Cenário                                                   | Objetivo                                                                                             | Pré-condições                                                 | Passos                                                                                 | Resultado Esperado                                                                                                                                            |
| :------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------- | :------------------------------------------------------------ | :------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **CT-AUD-09**<br>Ordenação cronológica (mais recente primeiro) | Validar que os logs são exibidos em ordem decrescente de timestamp (mais novos no topo).             | - Múltiplos registros de auditoria com timestamps diferentes. | 1. Acessar `/audit`.                                                                   | - O primeiro registro da tabela tem o timestamp mais recente.<br>- Registros mais antigos aparecem abaixo.                                                    |
| **CT-AUD-10**<br>Variante visual por tipo de ação              | Validar que cada `actionType` tem uma cor/estilo diferente no badge.                                 | - Registros de `CREATE_ORDER` e `UPDATE_STATUS` existem.      | 1. Visualizar a tabela de auditoria.                                                   | - `CREATE_ORDER`: badge verde (emerald).<br>- `UPDATE_STATUS`: badge azul (blue).<br>- `UPDATE_DELIVERY` / `UPDATE_TRANSPORT`: badge âmbar (amber, fallback). |
| **CT-AUD-11**<br>Paginação na auditoria                        | Validar paginação com 8 itens por página.                                                            | - 10+ registros de auditoria.                                 | 1. Navegar entre páginas.                                                              | - Paginação funciona corretamente com 8 itens por página.                                                                                                     |
| **CT-AUD-12**<br>Estado vazio - Nenhum log de auditoria        | Validar exibição de empty state.                                                                     | - `auditLogs` no localStorage está vazio.                     | 1. Acessar `/audit`.                                                                   | - Exibe ícone de inbox com mensagem descritiva.                                                                                                               |
| **CT-AUD-13**<br>Apenas uma linha expandida por vez            | Validar que ao clicar em "Inspecionar" em uma segunda linha, a primeira é automaticamente recolhida. | - Tabela com múltiplos registros.                             | 1. Clicar em "Inspecionar" no registro A.<br>2. Clicar em "Inspecionar" no registro B. | - Apenas o registro B fica expandido.<br>- O registro A é recolhido automaticamente (comportamento `expandedLogId` único).                                    |
